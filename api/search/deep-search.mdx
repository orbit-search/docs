Developer Deep Search starts asynchronous profile enrichment from an email or LinkedIn seed, or from a structured search intent, and returns IDs you can use to track progress. Send `email` and/or `linkedinUrl` for direct profile generation, or send the same public `structuredIntent` DSL used by Search for arbitrary discovery.

Each accepted run costs **10 credits**. Failed validation, failed enqueue, and other rejected requests do not consume credits.

## Authentication

Include your developer API key in every request. The key must have the `search:read` scope.

```http
Authorization: Bearer sk_orb_...
```

## Start a run

```http
POST /v2/developer/deep-search
```

### Headers

| Header | Required | Description |
|---|---|---|
| `Authorization` | Yes | `Bearer sk_orb_...` with `search:read` scope |
| `Content-Type` | Yes | `application/json` |
| `Idempotency-Key` | Recommended | Client-generated key for safe retries |

<Tip>
Send an `Idempotency-Key` for every new Deep Search request. Reusing the same key with the same body returns the existing run. Reusing it with a different body returns a `409`.
</Tip>

### Request body

Use email or LinkedIn when you already have a direct identity seed:

```json
{
  "email": "jane@example.com",
  "linkedinUrl": "https://www.linkedin.com/in/janedoe"
}
```

Use structured intent when you want Orbit to find matching people and generate profiles:

```json
{
  "structuredIntent": {
    "version": "v1",
    "names": ["Jai"],
    "experiences": [{ "organization": "Orbit Intelligence" }],
    "demographics": { "ageRange": { "min": 23, "max": 25 } }
  }
}
```

<ParamField body="email" type="string">
  Optional email address to use as a Deep Search generation seed. Send `email`, `linkedinUrl`, or both for profile generation.
</ParamField>

<ParamField body="linkedinUrl" type="string">
  Optional LinkedIn profile URL to use as a Deep Search generation seed. Send `linkedinUrl`, `email`, or both for profile generation.
</ParamField>

<ParamField body="structuredIntent" type="object">
  Structured search intent. `names[0]` is required as the person name. Experiences, schools, semantic clauses, geo text, and demographics become Deep Search discovery terms. `demographics.ageRange` maps to `age` or `age_range`; `demographics.birthYearRange` is converted into an approximate `age_range`.
</ParamField>

Structured discovery requires `structuredIntent.names[0]` plus at least one discovery signal that maps to search terms, `age`, or `age_range`.

<Note>
  Deep Search does not accept Search's `query` field or the legacy top-level fields (`name`, `web_url`, `search_terms`, `age`, `age_range`, and location fields). `entityClauses` are not part of the public structured DSL yet. Deep Search also does not have the synchronous search endpoint's API-key user network or location context, so use `POST /v2/developer/search` for strict `personalization.network` or `personalization.nearMe` behavior.
</Note>

Behind the API boundary, Orbit converts `structuredIntent` into the internal Deep Search arbitrary-discovery payload. Do not send internal Deep Search fields such as `person_name` or `arbitrary_search_terms` directly.

### 202 Accepted

Email/LinkedIn seed requests return `mode: "generation"`:

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "generation",
    "status": "processing",
    "stage": "queued",
    "search_id": "DEEP_SEARCH_ID",
    "job_id": "DEEP_SEARCH_JOB_ID",
    "cluster_group_id": null,
    "progress_url": "/v2/developer/deep-search/RUN_UUID",
    "events_url": "/v2/developer/deep-search/RUN_UUID/events",
    "result": {
      "profile_ids": []
    }
  }
}
```

Structured discovery requests return `mode: "arbitrary"`:

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "arbitrary",
    "status": "processing",
    "stage": "queued",
    "search_id": "DEEP_SEARCH_ID",
    "job_id": "DEEP_SEARCH_JOB_ID",
    "cluster_group_id": "CLUSTER_GROUP_ID",
    "progress_url": "/v2/developer/deep-search/RUN_UUID",
    "events_url": "/v2/developer/deep-search/RUN_UUID/events",
    "result": {
      "profile_ids": []
    }
  }
}
```

The response contains identifiers only. Use `progress_url` or `events_url` to track sources, candidates, generation stage, and final profile IDs.

## Get a progress snapshot

```http
GET /v2/developer/deep-search/:id
```

Snapshot reads require the same API key that created the run.

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "arbitrary",
    "status": "processing",
    "stage": "generating_candidates",
    "sources_unassigned": [],
    "candidates": [
      {
        "cluster_id": "cluster-1",
        "child_search_id": "child-search-1",
        "status": "processing",
        "stage": "generating_profile",
        "progress_percent": 75,
        "sources": [
          {
            "id": "source-1",
            "url": "https://example.com/jai",
            "title": "Jai at Orbit Intelligence",
            "snippet": "Public source excerpt",
            "source_type": "web"
          }
        ],
        "profile_ids": {
          "id": null,
          "orbit_id": null
        }
      }
    ],
    "result": {
      "profile_ids": []
    }
  }
}
```

Stable public stages are `queued`, `finding_sources`, `clustering_sources`, `dispatching_candidates`, `generating_candidates`, `generating_profile`, `saving_profile`, `completed`, and `failed`.

## Stream progress with SSE

```http
GET /v2/developer/deep-search/:id/events
```

The events endpoint emits `progress` events whenever the public snapshot changes, followed by a terminal `completed` or `failed` event.

```text
event: progress
data: {"status":"success","payload":{"id":"RUN_UUID","stage":"finding_sources"}}

event: completed
data: {"status":"success","payload":{"id":"RUN_UUID","status":"completed"}}
```

Reconnect with the same run ID to continue tracking progress. Already-terminal runs emit their terminal snapshot immediately. If the stream emits an `error` event or disconnects before a terminal event, keep the run ID and fall back to `GET /v2/developer/deep-search/:id`; snapshot reads are the recovery path for transient streaming or upstream progress-broker issues.

## Credits and rate limits

Each accepted Deep Search run costs **10 credits**. Snapshot and SSE reads are rate-limited but do not create new Deep Search runs or consume credits.

Deep Search uses the search rate-limit bucket: **60 requests per 60 seconds** per API key by default. This applies to start, snapshot, and events requests.

## Error responses

<AccordionGroup>
  <Accordion title="400 - Bad request">
    | Code | Description |
    |---|---|
    | `developer_deep_search_field_unsupported` | The body includes a top-level field other than `email`, `linkedinUrl`, or `structuredIntent` |
    | `developer_deep_search_input_required` | The body is missing `email`, `linkedinUrl`, or `structuredIntent` |
    | `developer_deep_search_email_invalid` | `email` is malformed |
    | `developer_deep_search_linkedin_invalid` | `linkedinUrl` is malformed or is not a LinkedIn profile URL |
    | `developer_deep_search_input_conflict` | The body includes both a profile generation seed and `structuredIntent` |
    | `developer_deep_search_name_required` | `structuredIntent.names[0]` is missing or empty |
    | `developer_deep_search_arbitrary_input_required` | Structured intent is missing a discovery signal that maps to search terms, `age`, or `age_range` |
    | `developer_deep_search_personalization_unsupported` | `structuredIntent.personalization` was provided, but Deep Search cannot honor user-context filters |
    | `structured_intent_required` | `structuredIntent` was provided but its value is not a valid object |
    | `structured_intent_invalid` | `structuredIntent` is malformed or contains unsupported values |
    | `structured_intent_version_unsupported` | `structuredIntent.version` must be `"v1"` |
    | `structured_intent_entity_clauses_unsupported` | `entityClauses` are not part of the public structured DSL |
    | `structured_intent_server_owned_field` | The request includes a server-owned structured intent field |
  </Accordion>
  <Accordion title="402 - Payment required">
    | Code | Description |
    |---|---|
    | `developer_api_credits_insufficient` | Your remaining credits are lower than the 10 credits required to start a run |
  </Accordion>
  <Accordion title="404 - Not found">
    | Code | Description |
    |---|---|
    | `developer_deep_search_run_not_found` | The run ID does not exist for this API key |
  </Accordion>
  <Accordion title="409 - Conflict">
    | Code | Description |
    |---|---|
    | `developer_deep_search_idempotency_key_conflict` | An explicit `Idempotency-Key` was reused with a different body |
    | `developer_deep_search_credit_ledger_conflict` | A request-level retry matched an existing credit reservation with different request parameters |
    | `developer_deep_search_existing_charge_without_run` | A prior retry charge exists but no matching run record was found |
  </Accordion>
  <Accordion title="429 - Rate limited">
    | Code | Description |
    |---|---|
    | `developer_api_key_rate_limited` | The API key exceeded the search rate limit |
  </Accordion>
  <Accordion title="502 - Upstream error">
    | Code | Description |
    |---|---|
    | `developer_deep_search_enqueue_failed` | Deep Search did not accept the async enqueue request |
  </Accordion>
</AccordionGroup>

## Code examples

Direct profile generation from email/LinkedIn seeds:

<CodeGroup>

```bash curl
curl -X POST "$API_BASE/v2/developer/deep-search" \
  -H "Authorization: Bearer sk_orb_REDACTED" \
  -H "Idempotency-Key: request-uuid-or-client-retry-key" \
  -H "Content-Type: application/json" \
  -d '{"email":"jane@example.com","linkedinUrl":"https://www.linkedin.com/in/janedoe"}'
```

```javascript JavaScript
const response = await fetch(`${API_BASE}/v2/developer/deep-search`, {
  method: "POST",
  headers: {
    "Authorization": "Bearer sk_orb_REDACTED",
    "Idempotency-Key": crypto.randomUUID(),
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    email: "jane@example.com",
    linkedinUrl: "https://www.linkedin.com/in/janedoe",
  }),
});

const data = await response.json();
console.log(data.payload.id, data.payload.mode);
```

```python Python
import uuid
import requests

response = requests.post(
    f"{API_BASE}/v2/developer/deep-search",
    headers={
        "Authorization": "Bearer sk_orb_REDACTED",
        "Idempotency-Key": str(uuid.uuid4()),
        "Content-Type": "application/json",
    },
    json={
        "email": "jane@example.com",
        "linkedinUrl": "https://www.linkedin.com/in/janedoe",
    },
)

data = response.json()
print(data["payload"]["id"], data["payload"]["mode"])
```

</CodeGroup>
