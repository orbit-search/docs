Developer Deep Search starts asynchronous profile enrichment from a singular identity seed, or candidate discovery from a structured search intent, and returns IDs you can use to track progress. Send `linkedinUrl`, `email + structuredIntent`, `urls + structuredIntent.names[0]`, or `phoneNumber + structuredIntent.names[0]` for profile generation. Use `structuredIntent` with `mode: "generation"` or `partialProfile: true` when structured intent should generate one profile. Send only `structuredIntent`, or set `mode: "candidate_discovery"`, when you want candidate discovery instead.

Each accepted run costs **10 credits**. Failed validation, failed enqueue, and other rejected requests do not consume credits.

## Authentication

Include your developer API key in every request. The key must have the `search:read` scope.

```http
Authorization: Bearer sk_orb_...
```

## Start a run

```http
POST /v2/developer/deep-search
```

### Headers

| Header | Required | Description |
|---|---|---|
| `Authorization` | Yes | `Bearer sk_orb_...` with `search:read` scope |
| `Content-Type` | Yes | `application/json` |
| `Idempotency-Key` | Recommended | Client-generated key for safe retries |

<Tip>
Send an `Idempotency-Key` for every new Deep Search request. Reusing the same key with the same body returns the existing run. Reusing it with a different body returns a `409`.
</Tip>

### Request body

Use LinkedIn when you already have a direct identity seed:

```json
{
  "linkedinUrl": "https://www.linkedin.com/in/janedoe"
}
```

Add `partialProfile: true` when you want the run to publish an early source-backed profile before the full Deep Search pipeline finishes:

```json
{
  "linkedinUrl": "https://www.linkedin.com/in/janedoe",
  "partialProfile": true
}
```

Use public URLs plus a structured name when the URL is the identity seed:

```json
{
  "urls": ["https://github.com/janedoe"],
  "structuredIntent": {
    "version": "v1",
    "names": ["Jane Doe"]
  },
  "partialProfile": true
}
```

Use structured intent alone when you want Orbit to discover matching people:

```json
{
  "mode": "candidate_discovery",
  "structuredIntent": {
    "version": "v1",
    "names": ["Jai"],
    "experiences": [{ "organization": "Orbit Intelligence" }],
    "semanticClauses": [
      { "text": "posts about consumer social startups" }
    ],
    "demographics": { "ageRange": { "min": 23, "max": 25 } }
  }
}
```

<ParamField body="mode" type="string">
  Optional mode. Use `"generation"` to enrich one profile, or `"candidate_discovery"` to find candidate people from `structuredIntent`. When omitted, direct identity inputs default to `"generation"` and `structuredIntent` alone defaults to `"candidate_discovery"`.
</ParamField>

<ParamField body="email" type="string">
  Optional email address to use as an identity hint. Email-only generation is not supported; send it with `structuredIntent` or `linkedinUrl`.
</ParamField>

<ParamField body="phoneNumber" type="string">
  Optional phone number to use as an identity hint. Phone generation requires `structuredIntent.names[0]`; send the public field as `phoneNumber`, not `phone`.
</ParamField>

<ParamField body="linkedinUrl" type="string">
  Optional LinkedIn profile URL to use as a Deep Search generation seed. LinkedIn alone is supported; `email`, `phoneNumber`, `urls`, `usernames`, and `structuredIntent` can also be included as hints.
</ParamField>

<ParamField body="urls" type="string[]">
  Optional public URLs to use as source seeds or hints. `urls + structuredIntent.names[0]` is a supported profile generation path; `urls` alone is not supported.
</ParamField>

<ParamField body="usernames" type="string[]">
  Optional usernames to use as identity hints. Usernames alone are not supported.
</ParamField>

<ParamField body="partialProfile" type="boolean">
  Optional fast partial profile flag for generation requests. When `true`, Deep Search publishes a source-backed partial profile and profile sections as soon as they are ready, then continues the full pipeline. This flag is not supported in `candidate_discovery` mode.
</ParamField>

<ParamField body="structuredIntent" type="object">
  Structured search intent. For profile generation, `names[0]` is the person name and other fields are used as matching hints. For candidate discovery, `names[0]` plus at least one discovery signal is required.
</ParamField>

Candidate discovery requires `structuredIntent.names[0]` plus at least one supported discovery signal, such as an experience, school, semantic clause, geo place, age range, or birth-year range. Gender is not a Deep Search discovery signal by itself.

<Note>
  Put freeform discovery text in `structuredIntent.semanticClauses`; do not send it as top-level clauses. `entityClauses` are not part of the public structured DSL yet. Use `POST /v2/developer/search` when you need personalized network or near-me search behavior.
</Note>

### 202 Accepted

Profile generation requests return `mode: "generation"`:

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "generation",
    "status": "processing",
    "stage": "queued",
    "search_id": "DEEP_SEARCH_ID",
    "job_id": "DEEP_SEARCH_JOB_ID",
    "cluster_group_id": null,
    "progress_url": "/v2/developer/deep-search/RUN_UUID",
    "events_url": "/v2/developer/deep-search/RUN_UUID/events",
    "result": {
      "profile_ids": []
    }
  }
}
```

Structured discovery requests return `mode: "arbitrary"`:

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "arbitrary",
    "status": "processing",
    "stage": "queued",
    "search_id": "DEEP_SEARCH_ID",
    "job_id": "DEEP_SEARCH_JOB_ID",
    "cluster_group_id": "CLUSTER_GROUP_ID",
    "progress_url": "/v2/developer/deep-search/RUN_UUID",
    "events_url": "/v2/developer/deep-search/RUN_UUID/events",
    "result": {
      "profile_ids": []
    }
  }
}
```

The response contains identifiers only. Use `progress_url` or `events_url` to track sources, candidates, generation stage, and final profile IDs.

## Get a progress snapshot

```http
GET /v2/developer/deep-search/:id
```

Snapshot reads require the same API key that created the run.

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "arbitrary",
    "status": "processing",
    "stage": "generating_candidates",
    "sources_unassigned": [],
    "candidates": [
      {
        "cluster_id": "cluster-1",
        "child_search_id": "child-search-1",
        "status": "processing",
        "stage": "generating_profile",
        "progress_percent": 75,
        "sources": [
          {
            "id": "source-1",
            "url": "https://example.com/jai",
            "title": "Jai at Orbit Intelligence",
            "snippet": "Public source excerpt",
            "source_type": "web"
          }
        ],
        "profile_ids": {
          "id": null,
          "orbit_id": null
        }
      }
    ],
    "result": {
      "profile_ids": []
    }
  }
}
```

Stable public stages are `queued`, `finding_sources`, `clustering_sources`, `dispatching_candidates`, `generating_candidates`, `generating_profile`, `partial_profile_ready`, `saving_profile`, `completed`, and `failed`.

### Partial profile snapshots

When a generation request includes `partialProfile: true`, snapshots and SSE progress events can include `result.partial_profile` while the run is still processing. The `partial_profile_ready` stage means the source-backed partial profile was generated. Check `result.partial_profile.profile_read` to see whether the partial profile sections are ready for profile-read queries.

```json
{
  "status": "success",
  "payload": {
    "id": "RUN_UUID",
    "mode": "generation",
    "status": "processing",
    "stage": "partial_profile_ready",
    "progress": {
      "percent": 70,
      "description": "Partial profile generated",
      "current_step": "partial_profile_ready"
    },
    "search_id": "DEEP_SEARCH_ID",
    "job_id": "DEEP_SEARCH_JOB_ID",
    "cluster_group_id": null,
    "sources_unassigned": [],
    "candidates": [],
    "result": {
      "profile_ids": [],
      "partial_profile": {
        "ready": true,
        "stage": "post_fast_partial_enrichment",
        "source_count_included": 10,
        "fun_fact_count": 18,
        "profile": {
          "id": "PROFILE_ID",
          "orbit_id": "ORBIT_PROFILE_ID",
          "full_name": "Jane Doe"
        },
        "profile_read": {
          "status": "queued",
          "published": true,
          "sections_queued": true,
          "sections_ready": false,
          "id": "PROFILE_ID",
          "orbit_id": "ORBIT_PROFILE_ID"
        }
      }
    }
  }
}
```

`profile_read.status` can be `queued`, `ready`, `partial_failure`, or `failed`. Treat `sections_ready: true` as the signal that the partial profile sections are available through profile read. After the full run completes, use the final `result.profile_ids`; the final profile replaces the partial data.

## Stream progress with SSE

```http
GET /v2/developer/deep-search/:id/events
```

The events endpoint emits `progress` events whenever the public snapshot changes, followed by a terminal `completed` or `failed` event.

```text
event: progress
data: {"status":"success","payload":{"id":"RUN_UUID","stage":"finding_sources"}}

event: completed
data: {"status":"success","payload":{"id":"RUN_UUID","status":"completed"}}
```

Reconnect with the same run ID to continue tracking progress. Already-terminal runs emit their terminal snapshot immediately. If the stream emits an `error` event or disconnects before a terminal event, keep the run ID and fall back to `GET /v2/developer/deep-search/:id`; snapshot reads are the recovery path for transient streaming or upstream progress-broker issues.

## Credits and rate limits

Each accepted Deep Search run costs **10 credits**. Snapshot and SSE reads are rate-limited but do not create new Deep Search runs or consume credits.

Deep Search uses the search rate-limit bucket: **60 requests per 60 seconds** per API key by default. This applies to start, snapshot, and events requests.

## Error responses

<AccordionGroup>
  <Accordion title="400 - Bad request">
    | Code | Description |
    |---|---|
    | `developer_deep_search_field_unsupported` | The body includes a top-level field other than `mode`, `email`, `phoneNumber`, `linkedinUrl`, `urls`, `usernames`, `partialProfile`, or `structuredIntent` |
    | `developer_deep_search_input_required` | The body is missing a supported Deep Search input |
    | `developer_deep_search_email_invalid` | `email` is malformed |
    | `developer_deep_search_phone_invalid` | `phoneNumber` is malformed |
    | `developer_deep_search_linkedin_invalid` | `linkedinUrl` is malformed or is not a LinkedIn profile URL |
    | `developer_deep_search_field_invalid` | `urls` or `usernames` is malformed |
    | `developer_deep_search_email_only_unsupported` | Email-only generation is not supported; provide `structuredIntent` or `linkedinUrl` |
    | `developer_deep_search_generation_input_required` | Generation mode is missing a supported profile generation input |
    | `developer_deep_search_generation_minimum_path_required` | Generation requires `structuredIntent`, `linkedinUrl`, `urls + structuredIntent.names[0]`, or `phoneNumber + structuredIntent.names[0]` |
    | `developer_deep_search_phone_name_required` | `phoneNumber` was sent without `structuredIntent.names[0]` |
    | `developer_deep_search_candidate_discovery_structured_intent_only` | `candidate_discovery` mode only supports `structuredIntent` |
    | `developer_deep_search_partial_profile_invalid` | `partialProfile` is present but is not a boolean |
    | `developer_deep_search_partial_profile_generation_only` | `partialProfile` was sent in `candidate_discovery` mode |
    | `developer_deep_search_name_required` | `structuredIntent.names[0]` is missing or empty |
    | `developer_deep_search_arbitrary_input_required` | Structured intent is missing a discovery signal that maps to search terms, `age`, or `age_range` |
    | `developer_deep_search_personalization_unsupported` | `structuredIntent.personalization` was provided, but Deep Search cannot honor user-context filters |
    | `structured_intent_required` | `structuredIntent` was provided but its value is not a valid object |
    | `structured_intent_invalid` | `structuredIntent` is malformed or contains unsupported values |
    | `structured_intent_version_unsupported` | `structuredIntent.version` must be `"v1"` |
    | `structured_intent_entity_clauses_unsupported` | `entityClauses` are not part of the public structured DSL |
    | `structured_intent_server_owned_field` | The request includes a server-owned structured intent field |
  </Accordion>
  <Accordion title="402 - Payment required">
    | Code | Description |
    |---|---|
    | `developer_api_credits_insufficient` | Your remaining credits are lower than the 10 credits required to start a run |
  </Accordion>
  <Accordion title="404 - Not found">
    | Code | Description |
    |---|---|
    | `developer_deep_search_run_not_found` | The run ID does not exist for this API key |
  </Accordion>
  <Accordion title="409 - Conflict">
    | Code | Description |
    |---|---|
    | `developer_deep_search_idempotency_key_conflict` | An explicit `Idempotency-Key` was reused with a different body |
    | `developer_deep_search_credit_ledger_conflict` | A request-level retry matched an existing credit reservation with different request parameters |
    | `developer_deep_search_existing_charge_without_run` | A prior retry charge exists but no matching run record was found |
  </Accordion>
  <Accordion title="429 - Rate limited">
    | Code | Description |
    |---|---|
    | `developer_api_key_rate_limited` | The API key exceeded the search rate limit |
  </Accordion>
  <Accordion title="502 - Upstream error">
    | Code | Description |
    |---|---|
    | `developer_deep_search_enqueue_failed` | Deep Search did not accept the async enqueue request |
  </Accordion>
</AccordionGroup>

## Code examples

Direct profile generation from a LinkedIn seed with an email hint:

<CodeGroup>

```bash curl
curl -X POST "$API_BASE/v2/developer/deep-search" \
  -H "Authorization: Bearer sk_orb_REDACTED" \
  -H "Idempotency-Key: request-uuid-or-client-retry-key" \
  -H "Content-Type: application/json" \
  -d '{"email":"jane@example.com","linkedinUrl":"https://www.linkedin.com/in/janedoe"}'
```

```javascript JavaScript
const response = await fetch(`${API_BASE}/v2/developer/deep-search`, {
  method: "POST",
  headers: {
    "Authorization": "Bearer sk_orb_REDACTED",
    "Idempotency-Key": crypto.randomUUID(),
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    email: "jane@example.com",
    linkedinUrl: "https://www.linkedin.com/in/janedoe",
  }),
});

const data = await response.json();
console.log(data.payload.id, data.payload.mode);
```

```python Python
import uuid
import requests

response = requests.post(
    f"{API_BASE}/v2/developer/deep-search",
    headers={
        "Authorization": "Bearer sk_orb_REDACTED",
        "Idempotency-Key": str(uuid.uuid4()),
        "Content-Type": "application/json",
    },
    json={
        "email": "jane@example.com",
        "linkedinUrl": "https://www.linkedin.com/in/janedoe",
    },
)

data = response.json()
print(data["payload"]["id"], data["payload"]["mode"])
```

</CodeGroup>
