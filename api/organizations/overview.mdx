Every Orbit account comes with a personal organization created automatically at signup. Personal organizations give you a private space for your own connections and searches. Workspace organizations are shared environments that support multiple members, teams, and enterprise features — they are the primary context for directory search, team-based access control, and API integrations built on behalf of an organization. Organizations own directories, sources, people, and grants; all resource IDs in directory and search APIs are scoped to an organization.

<Tip>
  **Personal key vs. organization key:**
  - Use a **personal API key** for personal search use cases and your own connections. Personal keys meter credits against you individually and cannot access directory corpora.
  - Use an **organization API key** for directory search, team-scoped search, and enterprise integrations. Organization keys meter credits against the organization and must match a directory grant to search directory corpora.
</Tip>

## Organization API keys

Organization API keys are scoped to a specific organization. They authenticate as the organization (not as a user), which allows them to perform directory search and other org-scoped operations without a user session. You create and manage organization API keys using a normal authenticated user session.

### Create an organization API key

`POST /v2/organizations/:organizationId/api-keys`

Creates a new API key scoped to the specified organization. The raw key is returned once in the `payload.key` field — store it immediately. Subsequent list responses include only `display_key` and other metadata, never the full key.

### Path parameters

<ParamField path="organizationId" type="string" required>
  The UUID of the organization that will own the key.
</ParamField>

### Request body

<ParamField body="name" type="string" required>
  A human-readable label for this key (for example, `"production search"`). Used to identify the key in management views.
</ParamField>

<ParamField body="scopes" type="string[]" required>
  The API scopes this key is authorized to use. For example, `["search:read", "profile:read"]`.
</ParamField>

<ParamField body="environment" type="string">
  An optional environment label such as `"production"` or `"staging"`. Used for organizational purposes in management views.
</ParamField>

<ParamField body="team_ids" type="string[]">
  An optional list of team UUIDs. When provided, the key is restricted to those teams and can only access directories granted to those teams. See [teams](/api/organizations/teams) for details.
</ParamField>

### Response fields

<ResponseField name="id" type="string">
  UUID of the newly created API key.
</ResponseField>

<ResponseField name="key" type="string">
  The raw API key value. This is only returned once at creation — store it immediately and treat it as a secret.
</ResponseField>

<ResponseField name="display_key" type="string">
  A non-secret representation of the key combining prefix and suffix, for example `sk_orb_a0186...55fd`. Use this in UI and support workflows to identify the key without exposing the raw value.
</ResponseField>

<ResponseField name="name" type="string">
  The label you provided at creation.
</ResponseField>

<ResponseField name="scopes" type="string[]">
  The scopes assigned to this key.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of when the key was created.
</ResponseField>

<CodeGroup>

```bash curl
curl -X POST "$API_BASE/v2/organizations/$ORGANIZATION_ID/api-keys" \
  -H "Authorization: Bearer $USER_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "production search",
    "scopes": ["search:read", "profile:read"],
    "environment": "production"
  }'
```

</CodeGroup>

Example response:

```json

  "payload": {
```
